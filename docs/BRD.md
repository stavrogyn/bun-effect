# BRD: Бизнес-требования для local-first приложения (текст, аудио, видео)

## 1. Цель документа

Определить бизнес-требования и минимально необходимую архитектуру для приложения с:
- текстовым чатом;
- голосовыми сообщениями и аудиозвонками;
- видеозвонками;
- local-first моделью данных и синхронизации.

Документ фиксирует ожидаемое поведение продукта, принципы синка, роли серверных компонентов и критерии приемки.

## 2. Область охвата

- Форматы коммуникации: текст, voice notes, аудиозвонки, видеозвонки.
- Режимы: online и offline (где применимо).
- Платформа синхронизации: local-first (oplog + локальное хранение + асинхронная репликация).

## 3. Бизнес-требования

### BR-1. Сообщения (текстовый чат)

**Требования:**
- Отправка и получение сообщений в каналах и DM.
- Статусы сообщений: `draft`, `queued`, `sent`, `delivered`, `read`, `failed`.
- Редактирование/удаление по бизнес-правилам (например, редактирование в течение 15 минут; удаление "для себя" и "для всех").

**Local-first реализация:**
- Каждое действие фиксируется как операция в oplog:
  - `MessageAdd`
  - `MessageEdit`
  - `MessageDelete`
- Локальное хранение: `OPFS` (снимок состояния + oplog + индексы).
- Синк: `Service Worker` держит один `WebSocket` на origin, отправляет/получает ops и рассылает изменения во все вкладки.
- Разрешение конфликтов:
  - текст сообщения: `LWW` (или "последний по Lamport");
  - реакции: `Counter`/`OR-Set`;
  - удаление: `tombstone` (логическое удаление).

### BR-2. Реакции, треды, упоминания

**Требования:**
- Реакции на сообщения.
- Треды и replies.
- Упоминания.
- Поиск по сообщениям.

**Local-first реализация:**
- Реакции: `OR-Set` (если допустима 1 реакция на пользователя) или `Counter` (если без ограничений).
- Поиск: локальный индекс (минимум по последним `N` сообщениям), серверный полнотекстовый поиск - опционально.

### BR-3. Вложения

**Требования:**
- Отправка изображений/файлов.
- Предпросмотр вложений.
- Ограничения размера/типа.
- Прогресс загрузки.

**Local-first реализация:**
- Файл записывается в `OPFS` чанками (`chunking`).
- Сообщение создается сразу, вложение догружается фоново.
- В оффлайне: `queued upload` с автоматическим возобновлением после восстановления сети.

### BR-4. Голосовые сообщения (voice notes)

**Требования:**
- Запись, прослушивание, отправка, отмена.
- Авто-пауза при звонке/потере фокуса (по настройке).
- Лимиты (например, 10 минут), настройки качества (`bitrate`).

**Local-first реализация:**
- Аудио сохраняется в `OPFS`.
- Сразу создается `VoiceNoteAdd` (metadata), аудиофайл загружается асинхронно.
- В оффлайне данные остаются локально, статус `queued`, отправка при reconnect.

### BR-5. Аудиозвонки (1:1 и групповые)

**Требования:**
- `Join/leave`, `mute/unmute`, выбор устройства.
- Индикатор качества соединения.
- Автовосстановление соединения (`reconnect`).
- Для групп: лимит участников, роли (`speaker`/`listener`).

**Local-first реализация (онлайн):**
- Медиа: `WebRTC`.
- Signaling: `WebSocket` (обмен `SDP`/`ICE`).
- Топология:
  - 1:1 может работать через `P2P`;
  - групповые звонки - через `SFU` (mesh не масштабируется).

**Local-first поведение при сбоях сети:**
- UI остается отзывчивым, показывается статус `Reconnecting...`.
- Если reconnect неуспешен в течение `X` секунд, пользователю предлагается fallback: "Записать голосовое вместо звонка".

### BR-6. Видеозвонки (1:1 и групповые)

**Требования:**
- Камера on/off.
- Демонстрация экрана.
- Смена устройства.
- Фон (опционально).
- Активный спикер, сетка участников, pin участника.
- Адаптивное качество (`degrade gracefully`).

**Local-first реализация:**
- `WebRTC` + `SFU` для групповых сценариев.
- Клиент хранит локально:
  - настройки устройств;
  - последний выбранный `bitrate`/ограничения;
  - UI state (pins/layout).
- При плохой сети:
  - снижение FPS/разрешения;
  - отключение видео с сохранением аудио;
  - переключение в `audio-only mode`.

### BR-7. Запись звонка (опционально)

**Требования:**
- Запись аудио/видео звонка.
- Хранение и доступ по ролям.
- Политика ретеншна.

**Local-first реализация:**
- Предпочтительно: серверная запись на стороне `SFU` для консистентности.
- Локальная запись (если нужна): сохранение в `OPFS` и последующая загрузка как вложение/voice note; учитывать высокие требования к объему хранения и пропускной способности.

### BR-8. Presence и Typing

**Требования:**
- Статусы `online/offline`.
- Индикатор набора (`typing`).
- `last seen` (опционально, с учетом настроек приватности).
- Presence не должен влиять на консистентность local-first данных.

**Реализация:**
- Presence/typing передаются как `ephemeral events`.
- Не пишутся в oplog.
- Доставка через `WebSocket` в реальном времени.

### BR-9. Доставка и Read Receipts

**Требования:**
- Статусы `delivered/read` на уровне сообщения.
- Батчинг receipts.
- Настройки приватности.

**Реализация:**
- Вариант 1: receipts как ops.
- Вариант 2 (предпочтительно): receipts как server-tracked метрика.
- Local-first вариант: `ReceiptSet(userId, msgId, state)` с правилом `LWW`.

## 4. Серверные компоненты (Bun + Effect)

Минимальный набор сервисов:
1. `Sync Gateway (WebSocket)` - прием/выдача ops, `ACK`, `dedup`.
2. `Signaling (WebSocket)` - обмен `SDP`/`ICE` для WebRTC.
3. `SFU` - групповые аудио/видео звонки (допустим отдельный сервис).
4. `TURN` - relay для сложных NAT/firewall сценариев.
5. `Media Storage (HTTP)` - загрузка и раздача voice notes и вложений.
6. `Auth` - токены, права на workspace/channel/call.

## 5. Критерии приемки

1. Оффлайн-текст: отправлено 20 сообщений без сети - после reconnect все синхронизировалось с консистентным порядком.
2. Две вкладки: реакция/редактирование попадает во вторую вкладку < 200 мс без участия сервера.
3. Voice notes: запись сделана оффлайн - после восстановления сети отправлена и появилась на другом устройстве.
4. Звонок: при отключении Wi-Fi на 10 секунд UI показывает reconnect, после восстановления сети сессия продолжается.
5. Групповой видео-звонок: 6+ участников стабильно работает через SFU.

## 6. Нефункциональные ожидания

- Устойчивость к кратковременным обрывам сети.
- Предсказуемые состояния UI при деградации качества соединения.
- Масштабируемость групповых звонков за счет `SFU`.
- Безопасное хранение медиа и корректная авторизация на доступ.

## 8. Глоссарий

- **WebRTC (Web Real-Time Communication)** - браузерная технология для аудио/видео связи (P2P и через серверные топологии).
- **SFU (Selective Forwarding Unit)** - сервер пересылки медиапотоков между участниками без полного микширования.
- **TURN (Traversal Using Relays around NAT)** - relay-сервер для сценариев, где прямое соединение невозможно.
- **STUN (Session Traversal Utilities for NAT)** - механизм определения публичных адресов/путей для установления соединения.
- **SDP (Session Description Protocol)** - описание параметров медиа-сессии (кодеки, форматы, режимы).
- **ICE (Interactive Connectivity Establishment)** - механизм поиска рабочего сетевого пути между участниками.
- **RTP (Real-time Transport Protocol)** - протокол передачи аудио/видео в реальном времени.
